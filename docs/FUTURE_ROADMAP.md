## 1. 현재 상태 요약

### 1.1 완료된 기능 (Sprint 1~8)

| Sprint | 내용 | 상태 |
|--------|------|------|
| Sprint 1 | Electron + React 기반 자막 입출력 시스템, 듀얼 모니터 자막 창, 속기사 패널 | 완료 |
| Sprint 2 | 자막 스타일링 (5종 프리셋, 실시간 프리뷰, 리사이즈 사이드바) | 완료 |
| Sprint 3 | Express 서버 + JWT 인증 (게스트/카카오), 세션 CRUD, Prisma + SQLite | 완료 |
| Sprint 4 | 세션 기반 협업 (참가/나가기/종료, 자막 로그, 내보내기) | 완료 |
| Sprint 5 | WebSocket 실시간 통신 (Socket.io, 브로드캐스트, 재연결) | 완료 |
| Sprint 6 | 대기 영역 + 교대 시스템 (operator/standby 역할, 교대 요청, 자동 승격) | 완료 |
| Sprint 8 | 안정화 + 릴리스 (51개 테스트, 에러 핸들링, 성능 최적화, 빌드 준비) | 완료 |

### 1.2 현재 기술 지표

| 지표 | 값 |
|------|-----|
| 테스트 수 | 51개 (서버 단위/통합 + 클라이언트 스토어) |
| 자막 지연 (로컬) | < 1ms |
| 자막 지연 (WebSocket) | ~50ms + 네트워크 RTT |
| 최대 속기사 수 | 4명 (세션당) |
| 자막 프리셋 | 5종 (기본/방송/강연/고대비/미니멀) |
| 인증 방식 | 게스트 + 카카오 OAuth (SSL 이슈 보류) |

### 1.3 알려진 제한 사항

| 제한 | 영향 | 우선순위 |
|------|------|----------|
| 카카오 OAuth SSL 인증서 문제 | 소셜 로그인 불가 (게스트로 대체 중) | 높음 |
| 커스텀 폰트 미지원 | 시스템 폰트만 사용 가능 | 중간 |
| 웹 클라이언트 없음 | Electron 설치 필수 | 높음 |
| SQLite 단일 서버 | 다중 인스턴스 불가 | 중간 |
| E2E 테스트 없음 | UI 통합 검증 수동 | 중간 |

## 2. 단기 계획 (1~2개월)

### 2.1 Sprint 7: 웹 클라이언트 구축

**목표:** Electron 설치 없이 웹 브라우저에서 자막 세션에 참여할 수 있는 웹 클라이언트를 구축한다.

**핵심 가치:** 자막을 "보는" 사람(청중, 방송 관계자)은 Electron 설치 없이 URL만으로 자막을 확인할 수 있다. OBS 브라우저 소스와 호환하여 방송 자막으로 즉시 활용 가능하다.

**구현 계획:**

```
Phase 1: 공용 컴포넌트 분리
  - [ ] Electron 의존 코드와 순수 React 코드 분리 (예상 복잡도: 높음)
  - [ ] 공용 레이어: stores, types, services (socketService, apiClient)
  - [ ] Electron 전용 레이어: IPC 호출, electronAPI 참조
  - [ ] 웹 전용 레이어: 브라우저 API, 라우팅

Phase 2: 웹 앱 구축
  - [ ] Vite 기반 React SPA 프로젝트 생성 (`web/` 디렉토리) (예상 복잡도: 중간)
  - [ ] 라우팅: /login, /lobby, /session/:id, /caption/:sessionId
  - [ ] /caption/:sessionId - 자막 전용 페이지 (스타일 적용, 읽기 전용)
  - [ ] OBS 브라우저 소스 호환 (투명 배경, 커스텀 CSS 지원)

Phase 3: 배포
  - [ ] 서버를 클라우드에 배포 (Vercel/AWS EC2) (예상 복잡도: 중간)
  - [ ] 환경 변수 분리 (개발/스테이징/프로덕션)
  - [ ] GitHub Actions CI/CD 파이프라인 구축
```

**기술적 고려 사항:**
- Electron과 웹에서 동일한 Zustand 스토어를 공유하되, IPC 레이어만 추상화한다.
- 웹 자막 페이지(`/caption/:sessionId`)는 WebSocket 수신 전용으로, 최소 번들 사이즈를 목표로 한다.
- OBS 브라우저 소스는 Chromium 기반이므로 CSS Custom Properties로 외부 스타일 주입을 지원한다.

### 2.2 카카오/네이버 OAuth 완성

**목표:** 소셜 로그인을 완전히 지원하여 게스트 로그인 의존도를 낮춘다.

```
Phase 1: 카카오 OAuth SSL 해결
  - [ ] Node.js SSL 인증서 체인 문제 진단 (예상 복잡도: 중간)
  - [ ] NODE_EXTRA_CA_CERTS 또는 커스텀 HTTPS Agent 적용
  - [ ] 카카오 개발자 콘솔 Redirect URI 재검증 (2026년 UI 기준)
  - [ ] 통합 테스트 작성

Phase 2: 네이버 OAuth 추가
  - [ ] 네이버 로그인 API 연동 (예상 복잡도: 중간)
  - [ ] User 모델 provider='naver' 지원 (이미 스키마 준비됨)
  - [ ] 로그인 페이지에 네이버 버튼 추가
  - [ ] 딥링크 콜백 처리 (카카오와 동일 패턴)
```

### 2.3 커스텀 폰트 업로드

**목표:** 사용자가 자신의 폰트 파일(.ttf, .otf)을 업로드하여 자막에 적용할 수 있게 한다.

```
Phase 1: 폰트 관리 시스템
  - [ ] 폰트 파일 업로드 API (multer + 파일 저장) (예상 복잡도: 중간)
  - [ ] 업로드된 폰트를 사용자별 디렉토리에 저장
  - [ ] Font Face API로 동적 폰트 로딩 (@font-face 런타임 주입)

Phase 2: UI 통합
  - [ ] StyleSettingsPanel에 폰트 업로드 버튼 추가 (예상 복잡도: 낮음)
  - [ ] 업로드된 폰트 목록 표시 + 선택
  - [ ] Caption Window에서 동적 폰트 로딩 지원
```

### 2.4 자동 교대 타이머

**목표:** 설정된 시간(예: 15분) 후 자동으로 교대를 알리거나 실행하여 속기사 피로를 관리한다.

```
Phase 1: 타이머 시스템
  - [ ] 서버 측 교대 타이머 관리 (세션별 설정) (예상 복잡도: 중간)
  - [ ] 교대 알림 이벤트: operator:timer-warning (1분 전 경고)
  - [ ] 자동 교대 옵션: 타이머 만료 시 다음 standby로 자동 전환

Phase 2: 교대 이력 대시보드
  - [ ] 교대 이력 DB 모델 추가 (OperatorHistory) (예상 복잡도: 중간)
  - [ ] 세션 내 교대 이력 타임라인 UI
  - [ ] 속기사별 송출 시간 통계
```

## 3. 중기 계획 (3~6개월)

### 3.1 AI 자동 자막 보조

**목표:** AI 기술을 활용하여 속기사의 입력 부담을 줄이고 자막 품질을 향상시킨다.

**핵심 가치:** 속기사가 음성을 듣고 수동 입력하는 기존 워크플로우에 AI를 보조 도구로 추가한다. AI가 초안을 생성하고 속기사가 수정/확인하는 하이브리드 모델을 목표로 한다.

```
Phase 1: STT(Speech-to-Text) 연동
  - [ ] 음성 입력 캡처 (Electron desktopCapturer 또는 마이크 입력) (예상 복잡도: 높음)
  - [ ] Web Speech API 또는 외부 STT API (Google/Whisper) 연동
  - [ ] STT 결과를 속기사 패널에 "제안 텍스트"로 표시
  - [ ] 속기사가 Tab/Enter로 제안 수락/수정

Phase 2: LLM 기반 보조
  - [ ] Claude/GPT API 연동 (예상 복잡도: 중간)
  - [ ] 실시간 맞춤법 교정 (빨간 밑줄 + 자동 수정 제안)
  - [ ] 문장 완성 제안 (자동완성 드롭다운)
  - [ ] 전문 용어 사전 기반 변환 (예: 의료/법률 용어)

Phase 3: 키워드 하이라이트
  - [ ] 실시간 키워드 감지 (사전 등록 또는 AI 추출) (예상 복잡도: 중간)
  - [ ] 자막 출력 시 키워드 색상 하이라이트
  - [ ] 키워드 목록 관리 UI
```

**기술적 고려 사항:**
- STT는 정확도보다 속도 우선. 속기사의 최종 확인이 있으므로 "보조" 역할에 충실한다.
- LLM API 호출은 비동기이며 자막 송출을 차단하지 않는다.
- API 비용 관리를 위해 요청 빈도를 제한한다 (문장 단위, 디바운스 적용).

### 3.2 PostgreSQL 마이그레이션 + 스케일링

**목표:** SQLite에서 PostgreSQL로 마이그레이션하여 다중 서버 인스턴스와 대규모 동시 접속을 지원한다.

```
Phase 1: DB 마이그레이션
  - [ ] Prisma 스키마 provider를 postgresql로 변경 (예상 복잡도: 낮음)
  - [ ] PostgreSQL 서버 구축 (Docker 또는 클라우드 매니지드)
  - [ ] 데이터 마이그레이션 스크립트 작성
  - [ ] 기존 SQLite 개발 환경은 유지 (환경 변수로 분기)

Phase 2: 서버 스케일링
  - [ ] Redis Pub/Sub 도입 (Socket.io adapter) (예상 복잡도: 높음)
  - [ ] 다중 서버 인스턴스 간 WebSocket 이벤트 동기화
  - [ ] 세션 메모리(sessionMembers Map)를 Redis로 이관
  - [ ] 로드 밸런서(Nginx/ALB) 설정 (sticky session 또는 Redis adapter)

Phase 3: 성능 모니터링
  - [ ] APM 도구 도입 (Datadog / New Relic) (예상 복잡도: 중간)
  - [ ] DB 쿼리 성능 모니터링 (Prisma Metrics)
  - [ ] WebSocket 연결 수/메시지 처리량 대시보드
```

### 3.3 역할 확장 + 접근 제어

**목표:** admin/manager 역할을 추가하고, 세션 접근 제어를 강화한다.

```
Phase 1: 역할 시스템 확장
  - [ ] User 모델에 role 필드 추가 (admin/manager/user) (예상 복잡도: 중간)
  - [ ] admin: 전체 시스템 관리, 사용자 관리
  - [ ] manager: 세션 생성/관리, 멤버 초대/강제퇴장
  - [ ] user: 세션 참가, 자막 입력 (기존과 동일)

Phase 2: 세션 접근 제어
  - [ ] 세션 비밀번호 기능 (생성 시 설정, 참가 시 입력) (예상 복잡도: 중간)
  - [ ] 초대 링크 (일회용 토큰 기반 URL)
  - [ ] 멤버 강제 퇴장 (manager 권한)
  - [ ] 세션 공개/비공개 설정

Phase 3: 세션 녹화/재생
  - [ ] CaptionLog에 타임스탬프 기반 재생 기능 (예상 복잡도: 높음)
  - [ ] 녹화 시작/종료 API
  - [ ] 재생 UI (타임라인 슬라이더, 배속 조절)
  - [ ] 녹화본 다운로드 (SRT/VTT 자막 파일 형식)
```

### 3.4 분석 대시보드

**목표:** 속기사와 관리자에게 작업 통계를 제공하여 팀 운영 효율성을 높인다.

```
Phase 1: 데이터 수집
  - [ ] 속기사별 타이핑 이벤트 로깅 (WPM, 정확도) (예상 복잡도: 중간)
  - [ ] 교대 이력 기록 (교대 시각, 소요 시간, 요청/자동 구분)
  - [ ] 세션 메트릭스 (총 자막 수, 평균 자막 길이, 총 시간)

Phase 2: 대시보드 UI
  - [ ] 속기사별 타이핑 속도 그래프 (시간대별 WPM) (예상 복잡도: 높음)
  - [ ] 세션별 자막 품질 지표 (수정 빈도, 평균 길이)
  - [ ] 교대 패턴 시각화 (타임라인 차트)
  - [ ] 팀 통계 요약 (총 세션 수, 총 자막 수, 활동 시간)

Phase 3: 리포트 생성
  - [ ] 주간/월간 리포트 자동 생성 (예상 복잡도: 중간)
  - [ ] PDF 다운로드
  - [ ] 이메일 발송 (선택적)
```

## 4. 장기 비전 (6개월~1년)

### 4.1 SaaS 전환

**목표:** 클라우드 기반 서비스로 전환하여 개인/팀/기업 사용자에게 구독형 서비스를 제공한다.

```
Phase 1: 클라우드 인프라
  - [ ] AWS/GCP 기반 프로덕션 인프라 구축
  - [ ] Kubernetes 또는 ECS 컨테이너 오케스트레이션
  - [ ] CDN (CloudFront) 정적 자산 배포
  - [ ] 자동 스케일링 (트래픽 기반)

Phase 2: 멀티 테넌시
  - [ ] 조직(Organization) 모델 추가
  - [ ] 팀 관리 (초대/역할/결제)
  - [ ] 조직별 데이터 격리
  - [ ] 커스텀 도메인 지원 (caption.mycompany.com)

Phase 3: 과금 시스템
  - [ ] 무료/프로/엔터프라이즈 티어 설계
  - [ ] 결제 연동 (Stripe/토스페이먼츠)
  - [ ] 사용량 기반 과금 (동시 접속 수, 저장 용량, AI 사용량)
  - [ ] 구독 관리 대시보드

가격 티어 (예시):
  무료        : 1 세션, 2명, 기본 프리셋, 웹 뷰어
  프로        : 무제한 세션, 4명, 커스텀 폰트, AI 보조, 녹화
  엔터프라이즈 : 무제한 인원, SSO, 전용 서버, SLA 보장
```

### 4.2 다국어 지원

**목표:** UI 다국어화와 실시간 번역 자막으로 글로벌 접근성을 확보한다.

```
Phase 1: UI 국제화 (i18n)
  - [ ] react-intl 또는 i18next 도입
  - [ ] 한국어/영어/일본어 기본 지원
  - [ ] 날짜/시간/숫자 로케일 처리
  - [ ] 에러 메시지 다국어화

Phase 2: 실시간 번역 자막
  - [ ] 번역 API 연동 (Google Translate / DeepL / Papago)
  - [ ] 원본 자막과 번역 자막 동시 표시 (듀얼 라인)
  - [ ] 번역 언어 선택 (한->영, 한->일, 한->중 등)
  - [ ] 자막 출력창에서 언어별 폰트/스타일 분리

Phase 3: 다국어 입력 지원
  - [ ] 다국어 IME 호환성 검증
  - [ ] RTL(아랍어/히브리어) 텍스트 방향 지원
  - [ ] 유니코드 특수 문자 처리
```

### 6.2 핵심 차별화 요소

**1. "실시간 + 협업 + 교대"의 삼위일체**

기존 자막 도구는 대부분 "자동 생성(AI/STT)" 또는 "후편집"에 초점을 맞추고 있다. Realtime Caption Studio는 **사람이 직접 입력하는 속기사 전문 도구**로서, 다음 시나리오에서 AI 기반 자막보다 우월한 정확도를 제공한다:

- 전문 용어가 많은 학술 발표/의료 컨퍼런스
- 음질이 좋지 않은 현장 생방송
- 여러 화자가 겹치는 토론/패널 세션
- 100% 정확도가 요구되는 법정 속기

**2. 무중단 교대 시스템**

속기사의 피로 관리는 장시간 자막 작업에서 핵심 과제이다. 본 시스템의 즉시 교대 메커니즘(50ms 이내)은 자막 끊김 없이 속기사를 교체할 수 있어, 기존에 "교대 시 수초간 자막 공백"이 발생하던 문제를 해결한다.

**3. 오프라인 + 온라인 하이브리드**

네트워크 없이도 로컬 모드로 동작하므로, 방송국/행사장 등 네트워크가 불안정한 환경에서도 안정적으로 사용할 수 있다. 네트워크가 가능한 환경에서는 원격 협업까지 지원하는 유연한 구조이다.

## 부록: 의사결정 기록

### 왜 Electron인가?

웹 앱으로는 불가능한 핵심 기능이 3가지 있다:
1. **듀얼 모니터 제어** - 두 번째 모니터에 자막 창을 자동 배치하는 것은 브라우저 API로 불가능하다.
2. **투명 오버레이 창** - `transparent: true`, `alwaysOnTop: true` 조합은 OS 네이티브 기능이다.
3. **오프라인 동작** - 네트워크 없이도 로컬 모드로 완전한 기능을 제공해야 한다.

웹 클라이언트는 "보는 사람"을 위한 확장이지, "입력하는 속기사"의 주 도구를 대체하지 않는다.

### 왜 SQLite에서 시작했는가?

- 해커톤 특성상 빠른 프로토타이핑이 최우선이다.
- SQLite는 설치/설정 제로 (파일 하나).
- Prisma가 DB 추상화를 제공하므로, 스키마 변경 없이 PostgreSQL로 마이그레이션 가능하다.
- 개발/테스트 환경에서 SQLite를 유지하여 CI 속도를 보장한다.

### 왜 WebSocket 디바운스를 50ms로 설정했는가?

- 속기사의 평균 타이핑 속도: 300~500 타/분 = 120~200ms/키 입력
- 50ms는 키 입력 간격보다 짧아 "거의 실시간"이면서, 네트워크 부하를 60% 이상 감소시킨다.
- 로컬 IPC는 디바운스 없이 즉시 전송하여, 속기사 자신의 자막 창은 지연 없이 표시된다.
